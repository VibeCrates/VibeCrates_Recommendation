## 개요

본 문서는 VibeCrates 추천 시스템을 웹 서비스로 출시하기 위한 22주 로드맵이다.

담당 역할: AI 개발자
  - 모델 학습 / 파인튜닝
  - 데이터 클렌징 및 학습 데이터 파이프라인
  - AI 추론 서버 (FastAPI) 구축 및 유지보수

서비스 형태: 웹사이트
  - 자연어 쿼리 → movie / music / book 크로스 도메인 추천
  - 사용자 피드백 수집 → 모델 반복 개선

## 할 일 전체 목록 (AI 개발자 범위)
================================================================================

  A. 모델 파인튜닝
  ─────────────────────────────────────────────────────────────────────────────
  A-1  재학습 결과 확인 (val loss 곡선, early stopping 동작 여부)
  A-2  evaluate.py 실행 → 도메인별 Recall@1/5/10, NDCG@10, MRR 수치화
  A-3  Stage 1 overfitting 근본 원인 분석
         후보: 배치 내 false negative 비율 높음 / 데이터 다양성 부족 /
               InfoNCE temperature 과소 (현재 0.07)
  A-4  개선 실험 선택 (우선순위)
         (1) In-batch hard negative mining 도입
         (2) Temperature 튜닝 (0.07 → 0.1~0.2)
         (3) 도메인별 샘플 가중치 조정
  A-5  DualEncoderModel config 파라미터 연동 리팩토링
         현재 lora_dropout이 config.lora_dropout에 연결되지 않고 hardcode
         DualEncoderModel(config) 형태로 변경 필요
  A-6  도메인별 성능 편차 분석 (movie / music / book 각각)
  A-7  피드백 데이터 기반 2차 fine-tuning (Phase 4에서)

  B. 데이터 클렌징
  ─────────────────────────────────────────────────────────────────────────────
  B-1  run_quality_test.py 실행 → 쿼리 DSV 불량률 정량화
         체크 항목: 유효 파이프(3분할), 한국어 잔존, 번호 prefix, 페르소나 레이블 잔존
  B-2  불량 쿼리 처리 전략 결정
         (a) threshold 이하 아이템 재생성 (Qwen 재호출)
         (b) 불량 행 필터링 후 데이터셋 재빌드
  B-3  이미지 누락·손상 아이템 처리
         data/books_invalid_indices.json 반영 → 해당 아이템 image fallback 정책 수립
         (옵션: 빈 텐서, 도메인 대표 이미지, 이미지 없이 text-only 경로)
  B-4  도메인별 중복 아이템 제거 (item_id 기준 dedup)
  B-5  train_dataset.csv 재빌드 및 버전 태깅 (v1 → v2)
  B-6  indexes/ 재생성 (클렌징 후 모델 재학습 시 연동)

  C. AI 서버 (FastAPI) 구축
  ─────────────────────────────────────────────────────────────────────────────
  C-1  E2E 플로우 테스트
         서버 시작 → load_index → POST /recommend 쿼리 → 결과 확인
  C-2  /feedback 엔드포인트 설계 및 구현
         스키마: { query, item_id, domain, feedback_type (click/like/dislike), timestamp }
         저장: append-only JSONL 또는 DB
  C-3  추론 레이턴시 측정 및 최적화
         목표: 쿼리 응답 < 200ms (GPU 서버 기준)
         후보 최적화: async encode_query, 배치 추론, torch.compile
  C-4  에러 핸들링 강화
         index 미로드, 모델 OOM, 타임아웃 케이스 HTTP 상태코드 표준화
  C-5  Docker 컨테이너화
         Dockerfile + docker-compose (AI 서버 + 볼륨 마운트)
         환경변수: MODEL_PATH, INDEX_DIR, DEVICE, IMAGE_DIR
  C-6  API 문서 정비 (OpenAPI 스펙 검토, 예시 쿼리 추가)
  C-7  모델 버전 관리 체계
         /recommend 요청에 model_version 필드 추가 → A/B 테스트 대비
  C-8  모니터링 기초 세팅
         추론 오류율, 평균 응답 시간, 도메인별 요청 수 로깅

  D. 서비스 연동
  ─────────────────────────────────────────────────────────────────────────────
  D-1  프론트엔드팀과 API 스펙 협의 및 확정
  D-2  통합 테스트 (프론트 ↔ AI 서버 ↔ indexes)
  D-3  레이턴시 SLA 합의 (권장: p95 < 300ms)
  D-4  부하 테스트 (동시 요청 시 VRAM / 응답 시간 측정)
  D-5  cold start 대응 정책 확정 (신규 사용자 초기 추천 전략)

  E. 사용자 설문 / 피드백 루프
  ─────────────────────────────────────────────────────────────────────────────
  E-1  피드백 데이터 스키마 최종 확정 (implicit / explicit 분리)
  E-2  수집 데이터 분석 파이프라인 구축
         피드백 → hard negative 추출 → train_dataset 보강 → re-training
  E-3  A/B 테스트 프레임워크 (기존 모델 vs re-trained 모델 비교)
  E-4  설문 결과 기반 2차 fine-tuning 실행 및 평가

## 주요 리스크 및 대응 방안

  R4  피드백 데이터가 W17까지 충분히 쌓이지 않는 경우 (< 100건)
      → 2차 fine-tuning을 W22 이후로 연기, 대신 합성 hard negative 활용
