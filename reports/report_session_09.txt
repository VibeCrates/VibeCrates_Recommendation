  ### 3-2. 스크립트 버그 발견

  [버그 1] 검증 실패 시 raw 저장
    기존: cache[item_id] = dsv if dsv else raw
    → validate_dsv 실패해도 쓰레기 출력이 CSV에 저장됨

  [버그 2] 완료 메시지 허위 보고
    기존: valid = df["query"].notna().sum()
    → 실제 DSV 유효성 검증 없이 notna()로 카운팅
    → "완료! 유효 쿼리: 40,109 / 40,109" 메시지가 실제 품질과 무관

  [버그 3] validate_dsv 허점
    → 다중 줄 출력에서 첫 번째 유효 줄만 통과시키는 방식
    → 모델이 "시인|...|..." 형태로 페르소나 레이블을 파트로 출력해도 통과


  ### 3-3. 세션 08 모델 선정 실험의 문제

  session_08에서 Qwen2.5-VL-7B 채택 근거였던 "90~100% 유효율"은
  아래 이유로 신뢰할 수 없음.

  - 테스트 샘플 수가 10건에 불과
  - test_qwen_queries.py (4페르소나)와 generate_queries.py (3페르소나)의
    프롬프트가 불일치 — 어느 스크립트로 측정했는지 불명확
  - 7B(bfloat16) vs 32B(4-bit) 비교 시 모델 크기와 양자화가 동시에 달라
    32B 탈락 원인이 크기인지 양자화인지 구분 불가

  [이번 세션 재검증 결과 — 50건 랜덤 테스트]
  항목                  Movie   Music   Book
  --------------------|-------|-------|------
  유효 DSV             90%     96%     94%
  영어 포함            28%     22%     20%
  파이프 분할 이상     24%     26%     16%
  페르소나 레이블      2%      4%      2%

  → "유효 DSV"도 validate_dsv 허점으로 실제보다 높게 측정됨

## 4. generate_queries.py 수정 사항
================================================================================

  [수정 1] 체크포인트 주기 단축
    CHECKPOINT_EVERY: 500 → 50
    이유: 500건 전에 프로세스 종료 시 모든 진행분 유실 방지

  [수정 2] 실패 시 raw 저장 제거
    변경 전: cache[item_id] = dsv if dsv else raw
    변경 후: if dsv: cache[item_id] = dsv
    효과: 검증 실패 건이 캐시에 없으므로 재실행 시 자동 재처리

  [수정 3] 완료 메시지 카운팅 수정
    변경 전: df["query"].notna().sum()
    변경 후: len(cache)
    효과: 실제 유효 쿼리 수 + 실패 건 수 정확히 보고

  [미수정] validate_dsv 로직, 프롬프트
    - 네거티브 예시 추가 프롬프트 테스트 결과 오히려 악화
      (Music 페르소나 레이블 4% → 54%)
    - 기존 프롬프트로 복구하여 진행 결정
    - validate_dsv 수정은 이번 세션 범위 외로 보류
