# VibeCrates 화면 명세 — 메인 페이지 · 보드 페이지

작성일: 2026-07-10
대상: `mockups/vibe_crate/` 디자인 시안 (생성: `scripts/build_vibe_mockup.py`)

---

## 1. 개념 정의

**보드(Board)**: 검색어 하나로 만들어진 추천 결과 묶음. 제목은 항상 사용자가
입력한 검색어와 동일하다. 보드는 movie/music/book 아이템 여러 개를
콜라주 형태로 담고 있다.

**Crates**: 서비스 이름이자 메인 페이지 타이틀. 메인 페이지는 여러 보드를
모아 보여주는 진입점이며, 동시에 검색으로 새 보드를 만드는 곳이다.

화면은 두 종류뿐이다: **메인 페이지**(보드 목록 + 검색) / **보드 페이지**(보드
상세, 콜라주 + 액션).

---

## 2. 메인 페이지 (`main.html`)

### 구성 (위→아래)

1. 사이트 타이틀 "Crates" — 중앙 정렬, 상단
2. 검색창 — 중앙 정렬, placeholder는 회색 이탤릭 `create your own collection via searching`
3. 보드 그리드 — 3개 섹션, 섹션마다 4열 그리드
   - **오늘의 추천 보드** (`section: "recommended"`)
   - **내 보드** (`section: "mine"`)
   - **인기 보드** (`section: "popular"`)

### 보드 타일

- 실제 보드의 콜라주를 축소한 썸네일 (`board.html`과 동일한 렌더 함수를
  1:1 정사각형 컨테이너에 그려서 만듦 — 별도 썸네일 이미지를 생성하지 않음)
- 썸네일은 hover 카드·링크가 비활성화된 순수 이미지 표시 (`interactive: false`)
- 타일 클릭 → `board.html?id={보드id}`로 이동
- 타일 아래 보드 제목 캡션

### 검색 동작

검색창에 검색어 입력 후 제출:
1. 새 보드 객체 생성 (`id`, `title=검색어`, `section: "mine"`, `items`)
2. `localStorage`에 저장
3. 방금 만든 보드 페이지로 이동

> **현재 한계**: 아직 추천 엔진이 연결되어 있지 않아, `items`는 기존
> 아이템 풀을 무작위로 섞어서 채운 placeholder다. 실제로는 이 시점에
> 추천 API를 호출해 검색어에 맞는 아이템을 받아와야 한다.

"내 보드" 섹션은 데모용 고정 보드 4개 + 이렇게 새로 만들어졌거나 보드
페이지에서 저장된 보드 전체를 합쳐서 보여준다.

## 3. 보드 페이지 (`board.html`)

### 구성

```
┌──────────────────────────────────────────────────────┐
│ [← 나가기]     [   검색창 (중앙, sticky)   ]  [저장][캡쳐]│  ← 항상 화면 상단 고정
├──────────────────────────────────────────────────────┤
│                                                        │
│               보드 제목 (H1 = 검색어)                    │
│                                                        │
│              ┌─── 콜라주 캔버스 ───┐                    │
│              │  (12개 아이템, 겹치고 회전)  │              │
│              └────────────────────┘                    │
└──────────────────────────────────────────────────────┘
```

- 헤더는 `position: sticky`로 스크롤해도 항상 보임
- 어떤 보드를 보여줄지는 URL `?id=` 쿼리 파라미터로 결정
  (`boards.js`의 고정 보드 또는 `localStorage`에 저장된 커스텀 보드 조회)

### 콜라주 캔버스

- 8개 존(zone)에 최대 12개 아이템을 겹치고 회전시켜 배치 (좌표는
  `common.js`의 `SLOTS` 배열, 퍼센트 기반이라 화면 크기에 맞춰 스케일됨)
- 이미지는 항상 보임 (기본 상태)
- 카드에 마우스 오버 시 하단에서 어두운 그라디언트 오버레이가 뜨며
  **`domain → title → year → url` 순서로 딜레이(0/50/100/150ms)를 두고
  차례로 나타남**
- `url`은 실제 클릭 가능한 링크 (새 탭)

### 검색창

- 항상 화면 상단에 존재 (sticky), 현재 보드 제목이 기본값으로 채워져 있음
- 제출 시 현재는 페이지 제목만 텍스트로 갱신 (실제 재추천 API 연동 전
  임시 동작)

### 액션 버튼

| 버튼 | 동작 |
|---|---|
| **내 컬렉션에 저장** | 클릭 시 `localStorage`에 저장 상태 토글. 라벨이 "내 컬렉션에 저장" ↔ "저장됨 ✓"으로 바뀜 |
| **캡쳐하기** | html2canvas(CDN)로 제목+콜라주 영역을 PNG로 캡처해 다운로드. 인터넷 연결 필요한 유일한 기능 |
| **나가기** | 보드가 저장 안 된 상태면 확인 모달 "보드를 저장하지 않고 이동하시겠습니까?" 표시 (취소/그냥 나가기). 저장된 상태면 모달 없이 바로 메인 페이지로 이동 |

## 4. 데이터 모델

### 아이템 (콜라주 조각 하나)

```js
{
  "id": "114709",            // 비노출. 이미지 파일 매핑 + DOM key 전용
  "title": "Toy Story",
  "domain": "movie",          // "movie" | "music" | "book"
  "year": 1995,               // 결측 시 null → 화면엔 "xxxx"
  "image": "images/movie/114709.jpg",
  "url": "http://www.imdb.com/title/tt0114709"
}
```

| 필드 | 노출 여부 | 출처 |
|---|---|---|
| `id` | 비노출 | movie=imdbId, music=spotify id, book=asin |
| `domain` | hover 시 | DB 스키마에 없음 — merge 시 백엔드가 직접 채움 |
| `title` | hover 시 | Title / name / title |
| `year` | hover 시 | release_date / year / publishedDate 에서 연도만 추출 |
| `image` | 항상 | **원본 DB의 Poster/img/imgUrl(외부 URL) 아님** — 로컬 이미지 `/Users/hyun/images/{domain}/{id}.jpg`를 id로 매핑 |
| `url` | hover 시 | Imdb Link / Spotify 트랙 URL 조립 / productURL |

### 보드 (아이템들을 묶고 제목=검색어를 가짐)

```js
{
  "id": "b1",
  "title": "비 오는 날 필요한 조합",    // = 사용자의 검색어
  "section": "recommended" | "mine" | "popular",
  "items": [ {...}, {...}, ... ]
}
```

`main.html`, `board.html` 모두 `boards.js`가 정의하는 전역 `BOARDS` 배열을
읽는다. `var`로 선언해야 `window.BOARDS`로 접근 가능 (`const`/`let`은 전역
객체 프로퍼티가 되지 않음).

## 5. 저장 방식 (지금은 프론트 전용)

로그인/백엔드가 없는 시안 단계라, "저장"과 관련된 모든 상태는
브라우저 `localStorage`에만 존재한다 (`common.js`):

| 용도 | 키 | 비고 |
|---|---|---|
| 저장한 보드 id 목록 | `vibecrates:savedBoardIds` | "내 컬렉션에 저장" 버튼이 토글 |
| 검색으로 만든 커스텀 보드 | `vibecrates:board:{id}` | 메인 페이지 검색 제출 시 생성 |

실제 서비스에서는 둘 다 로그인 유저 계정에 귀속되는 백엔드 API 호출로
대체되어야 한다 (지금 구조는 "그 자리에 무엇이 들어가야 하는지"를 보여주는
용도).

## 7. 아직 미정 — 백엔드/기획 논의 필요

- [ ] 보드 하나당 아이템 개수가 12개보다 적거나 많을 때 레이아웃 처리
- [ ] `section`(추천/내 보드/인기) 분류 기준 — 추천 알고리즘 점수 vs 저장
      횟수 집계 vs 그 외

      기준인지
